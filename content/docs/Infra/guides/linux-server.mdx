## Configuration Management

### Etckeeper

<Callout type="info">
We use `etckeeper` with Git backend on as many servers as possible for configuration tracking.
</Callout>

<Steps>
  <Step>
    **Before Changes**
    
    ```bash title="Save Current State"
    git status
    etckeeper commit "save before changes"
    ```
  </Step>
  
  <Step>
    **Make Changes**
    
    Perform your configuration modifications in `/etc`.
  </Step>
  
  <Step>
    **After Changes**
    
    ```bash title="Commit Changes"
    etckeeper commit "<descriptive message>"
    ```
  </Step>
</Steps>

<Callout type="tip">
See [etckeeper documentation](https://etckeeper.branchable.com/README/) for more details.
</Callout>

## Firewall Management

### Iptables Configuration

<Callout type="info">
We use iptables on many servers (generally host servers) with iptables-persistent for rule persistence.
</Callout>

<Tabs>
  <Tab value="ovh" label="OVH Servers">
    ### OVH Server Rules
    
    Rules are stored in:
    ```bash title="Rule Locations"
    /etc/iptables/rule.v4   # IPv4 rules
    /etc/iptables/rule.v6   # IPv6 rules
    ```
  </Tab>
  
  <Tab value="off1" label="OFF1 Server">
    ### OFF1 Server Rules
    
    Rules are stored in:
    ```bash title="Rule Location"
    /etc/iptables.up.rules
    ```
  </Tab>
</Tabs>

### Docker and Iptables

<Callout type="warning">
Docker has its own chains that are not affected by `INPUT` and `OUTPUT` rules. It won't block ports exposed by Docker.
</Callout>

<Steps>
  <Step>
    **Understanding Docker Chains**
    
    Docker creates its own iptables chains that bypass standard INPUT/OUTPUT rules.
  </Step>
  
  <Step>
    **Use DOCKER-USER Chain**
    
    To block Docker-exposed ports, add rules to the `DOCKER-USER` chain.
  </Step>
  
  <Step>
    **Reference Documentation**
    
    See [Docker iptables documentation](https://docs.docker.com/network/iptables/) for details.
  </Step>
</Steps>

## Shell Configuration Issues

### Root Bash Configuration

<Callout type="note">
Root is often created before autocompletion packages are installed, leading to limited shell features.
</Callout>

<Steps>
  <Step>
    **Check Current Configuration**
    
    Compare current and default configurations:
    ```bash title="Compare Bashrc"
    diff $HOME/.bashrc /etc/skel/.bashrc
    ```
  </Step>
  
  <Step>
    **Copy Default Configuration**
    
    If different, copy the default:
    ```bash title="Update Bashrc"
    cp /etc/skel/.bashrc /root/
    ```
  </Step>
</Steps>

## Troubleshooting Common Issues

<Tabs>
  <Tab value="color" label="No Color in Shell">
    ### Shell Color Issues
    
    <Steps>
      <Step>
        **Check TERM Variable**
        
        ```bash title="Check Terminal Type"
        echo $TERM
        ```
        
        Should be `xterm-256color` or `linux`.
      </Step>
      
      <Step>
        **Fix Bashrc**
        
        ```bash title="Compare and Fix"
        diff $HOME/.bashrc /etc/skel/.bashrc
        # If different, copy from /etc/skel/.bashrc
        ```
      </Step>
    </Steps>
  </Tab>
  
  <Tab value="completion" label="No Autocompletion">
    ### Autocompletion Issues
    
    <Steps>
      <Step>
        **Check bash-completion Package**
        
        ```bash title="Verify Installation"
        dpkg --verify bash-completion && echo ok
        ```
      </Step>
      
      <Step>
        **Fix Bashrc Configuration**
        
        ```bash title="Compare and Fix"
        diff $HOME/.bashrc /etc/skel/.bashrc
        # If different, copy from /etc/skel/.bashrc
        ```
      </Step>
    </Steps>
  </Tab>
</Tabs>
